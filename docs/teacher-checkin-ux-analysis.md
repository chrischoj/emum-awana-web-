# 교사 체크인 UX 분석

## 현재 구조

### 자동 체크인 (교사용 대시보드)
- 교사가 대시보드에 접속하면 자동으로 출석 처리됨
- GPS/위치 기반이 아닌 단순 접속 기반
- 피드백: 체크인 완료 시 토스트 알림 표시

### 관리자 출석부 (TeacherAttendancePage)
- 관리자가 교사별로 수동 토글 가능
- 날짜별 조회 및 클럽별 필터링 지원
- 비고란을 통해 부재 사유 등 기록 가능

## 현재 문제점

### 1. 이중 테이블 구조
- `teachers` 테이블과 `teacher_attendance` 테이블이 분리되어 있어
  교사 목록 로드 + 출석 데이터 로드의 2회 쿼리 필요
- 장점: 정규화된 구조로 데이터 무결성 보장
- 단점: JOIN 없이 클라이언트에서 매핑 필요

### 2. 미배정 교사 처리
- `club_id`가 null인 교사가 "그 외" 필터에 포함됨
- 미배정 교사도 출석 체크 대상인지 정책적 결정 필요
- 현재는 모든 active 교사가 출석 대상

### 3. 일괄 처리 부재 (해결됨)
- ~~교사를 한 명씩 토글해야 하는 비효율~~ → 전체 출석/해제 기능 추가로 해결

### 4. 자동 체크인 신뢰성
- 대시보드 접속 = 출석으로 처리하므로 실제 훈련 참석 여부와 괴리 가능
- 잠깐 접속 후 이탈해도 출석 처리됨
- 관리자가 수동으로 보정할 수 있는 구조이므로 현재로서는 수용 가능
